# Billuz — резервное копирование БД (PostgreSQL)

Защита от потери данных при разрыве сессии / отключении электричества: непрерывная
архивация журнала транзакций (WAL) + регулярные снимки БД. Восстановление возможно
**на любую секунду** (PITR) и **по тирам** минуты/часы/дни (снапшоты).

## Что куда сохраняется

Всё пишется в папку `BACKUP_HOST_DIR` (по умолчанию `D:\billuz-backups`, задаётся в корневом `.env`):

```
D:\billuz-backups\
  wal-archive\     ← WAL, архивируется Postgres непрерывно (основа PITR — любая секунда)
  basebackups\     ← физические базовые копии (pg_basebackup), якорь для PITR (ежедневно + при старте)
  minutely\        ← снапшоты pg_dump каждую минуту (хранится последние 60)
  hourly\          ← снапшоты каждый час (последние 48)
  daily\           ← снапшоты каждый день (последние 14)
  manual\          ← ручные снапшоты "сохранить сейчас"
  logs\backup.log  ← лог сайдкара
```

Тиры/ротация настраиваются в корневом `.env` (`MINUTELY_KEEP`, `HOURLY_KEEP`, `DAILY_KEEP`, `BASEBACKUP_KEEP`).

## Как это устроено

- **Postgres** (`billiard_pg`) запущен с `archive_mode=on` — каждый WAL-сегмент копируется в `wal-archive\`. Плюс `archive_timeout=60` — даже на простое сегмент сбрасывается раз в минуту.
- **Сайдкар** (`billiard_backup`) — отдельный контейнер с `scheduler.sh`: делает снапшоты по тирам и `pg_basebackup`. Работает, пока запущен Docker (`restart: unless-stopped`).

## Запуск

```bash
docker compose up -d postgres backup     # поднять БД с архивацией + сайдкар
# или просто: npm run backup:up
```

## Сохранение (save)

Снапшоты делаются автоматически. Ручной снапшот сейчас:

```bash
npm run backup:now            # → manual\billiard_<ts>.dump
npm run backup:list           # показать все доступные копии + диапазон WAL
npm run backup:logs           # лог сайдкара
```

## Восстановление (restore)

### Вариант A — из снапшота (быстро, повседневно; точность ~минута)

```bash
npm run backup:list                                   # выбрать файл
npm run restore:snapshot -- minutely/billiard_20260624T120000Z.dump
```
Пересоздаёт БД из выбранного дампа. Перед этим лучше остановить API.

### Вариант B — PITR на точный момент (точность до секунды)

```bash
# Git Bash, из корня репозитория:
ops/pg-backup/restore-pitr.sh "2026-06-24 15:30:00+05"
```
Берёт последнюю базовую копию и проигрывает WAL до указанного времени (`+05` = время Ташкента).
Останавливает контейнеры, перестраивает кластер, запускает Postgres — он доходит до цели и
открывается на запись. Прогресс: `docker logs -f billiard_pg`.

## Перенос на внешний/USB/сетевой диск

Сейчас копии на `D:`. Для настоящей защиты (диск может умереть вместе с БД) укажи другой носитель:
поменяй `BACKUP_HOST_DIR` в корневом `.env` на путь USB/сетевого диска и пересоздай контейнеры:
```bash
docker compose up -d postgres backup
```

## Важно про отключение питания

Зафиксированные (committed) транзакции Postgres и так переживают отключение питания
(WAL + fsync). Это резервирование добавляет защиту от порчи диска, ошибочного удаления и
даёт точки восстановления во времени.
